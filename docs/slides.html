What Was Built
GPT-4.1 Vision ingest in batches of five pages per call, no OCR dependency
Per-document wiki: pre-extracted summaries, entities, metrics and claims per section
Three swappable retriever modes: Naive vector, ISD with reranking, Wiki-first
ISD agent loop: decompose question, gather evidence, draft, verify, revise
Streaming React UI with cell state machine, PDF viewer and citation links
Current Limitations
Local only, single user, no authentication or workspace isolation
Wiki build cost of roughly $2 to $5 per 10-K filing needs amortisation strategy
Citations are page-level only, no sub-page bounding box highlights in the viewer
Charts, images and diagrams inside PDFs are not extracted or reasoned over
No document versioning, re-uploading a revised filing rebuilds from scratch
To Scale This
Cloud object storage for PDFs and vector indices
Multi-tenant auth with workspace isolation
Managed async job workers with priority queues and retry
Wiki build cost shared across users via content-addressable cache
Where It Fits
Analysts running deep due diligence across large document sets
Credit, equity research, and compliance review workflows
Any team repeating the same extraction questions across many filings
The Bigger Play
Matrix output becomes a structured custom database per client
That database becomes the backend for an MCP chat interface
Analysts query their own knowledge base conversationally, every answer traced to source
System Architecture
How a PDF becomes a cited, verified cell answer
Offline Pipeline  ·  Runs Once Per Document  ·  Cached by SHA-256
GPT-4.1 Vision Ingest Renders pages to PNG at 150 DPI
Sends batches of 5 images per API call
Returns markdown per page parser/pdf.py
→
Wiki Builder One LLM call per section
Extracts entities, claims, metrics, Q&A pairs
Doc-level rollup with key metrics table wiki/builder.py
→
Vector Index 500-token chunks with 50-token overlap
Azure text-embedding-3-large or BGE local
Stored in LanceDB, embedded on disk retriever/index.py
→
SQLite Store Workspaces, grids, columns, rows, cells
Job queue persisted across restarts
WAL mode for concurrent reads storage/models.py
Online  ·  Runs Per Cell  ·  One CellJob per (document, column) pair
Decompose Breaks column prompt into 2 to 4 sub-questions and identifies target sections from wiki index
→
Gather Evidence Runs retriever per sub-question, deduplicates by chunk ID, pools up to 18 evidence chunks
→
Draft Answer LLM writes answer from evidence only. Every factual claim must cite at least one chunk ID
→
Verify Claims Re-retrieves each claim, asks LLM whether evidence supports, contradicts or is missing it
→
Cell Result Answer, citations with page numbers, confidence level, full reasoning trace, tokens used
Retriever Mode Naive   vector top-k only ISD   decompose + LLM rerank Wiki   structured knowledge first Swappable at runtime. Benchmarked against FinanceBench.
FastAPI  +  Server-Sent Events Streams cell state changes in real time. Reconnect-safe via Last-Event-ID replay. Jobs continue server-side on disconnect.
→
React Frontend  ·  localhost:5173 TanStack Table grid, Zustand state, PDF.js viewer with citation overlays, Framer Motion cell animations, command palette.
