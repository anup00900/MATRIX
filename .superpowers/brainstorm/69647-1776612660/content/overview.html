Hebbia Matrix POC — How It Works

A plain-English walkthrough with block diagrams: what the matrix is, how the pipeline runs, what every prompt does.

① What is "The Matrix"?

Imagine a spreadsheet where rows = PDF documents and columns = questions you want answered. Each cell gets filled automatically by AI, with the answer and the exact page + snippet it came from.

Document Revenue YoY change? Main risk factors? Who is the auditor?
🟢 Apple 10-K FY23 +2.8% p.42 cited Supply chain, macro headwinds p.18 cited Ernst & Young p.91 cited
🔵 NVIDIA 10-K FY24 ⟳ streaming… — —
⚪ Netflix 10-K FY23 — — —
Why is this hard? A 10-K filing is 200+ pages. "Revenue YoY change" could be in the Income Statement, the MD&A section, or a footnote. The system must find the right passage, read it correctly, and show you where it came from — for every cell, for every document.

② The 4-Stage Pipeline

Stages 1–3 run offline (once per PDF, cached forever). Stage 4 runs online (every time a cell is filled).

Content-addressable caching: Every PDF is identified by its SHA-256 hash. If you upload the same Apple 10-K twice, the wiki is built only once. Same for embeddings.

Batch prompt sent to GPT-4.1 Vision (5 page images per call):

// BATCH VISION PROMPT (parser/pdf.py → BATCH_SYSTEM) You are a PDF page-to-markdown converter for financial filings. You will receive multiple consecutive page images, each labelled 'Page N:'. For each page convert to clean GitHub-flavored markdown: preserve heading hierarchy (#, ##, ###) render tables as markdown tables with aligned columns keep all numeric values and units exactly as printed preserve bullet/numbered lists Do NOT add commentary or invent content not visible in the images. Return ONLY a JSON array, one object per page in input order: [{"page_no": <int>, "markdown": "<extracted markdown>"}] Page 1: [image] Page 2: [image] Page 3: [image] Page 4: [image] Page 5: [image]

After all batches complete, meta extracted from first 3 pages of markdown:

// META EXTRACTION PROMPT (parser/meta.py) Extract the issuer name, filing type, and period_end date from the following filing header. Return JSON fields: company, filing_type, period_end. Unknown fields → null. [markdown from pages 1–3]
No OCR. All extraction is done by GPT-4.1 Vision — scanned pages, complex tables, multi-column layouts all handled by the model. If a batch API call fails, affected pages are marked failed=True with empty markdown; no silent fallback.

④ Stage 2 — WIKI BUILD (The "Pre-Digested Brain")

Instead of searching raw text at query time, the system pre-reads every section and extracts structured knowledge. Think of it as building a Wikipedia article about each section of the filing.

Per-Section Prompt (run for every section)
You are analysing a section of a financial filing. Section title: {section.title} Chunks: [up to 12 chunks, 600 chars each] Extract: - summary: 3-5 sentence summary - entities: named entities + chunk refs - claims: factual claims + evidence chunks - metrics: numbers with unit/period/chunk - questions_answered: what this section can answer
Doc-Level Rollup Prompt (one call per document)
Given these section summaries, write a 3-5 sentence overview, then return JSON: - overview: document-level summary - key_metrics_table: top 3-6 metrics (revenue, net_income, operating_margin…) [section summaries listed here]

The wiki is cached as storage/wikis/{sha256}__v1.json.gz. On schema change, it is lazily rebuilt — old files kept for reproducibility.

⑤ Stage 4 — ISD LOOP (How Each Cell Gets Filled)

"ISD" = Iterative Source Decomposition. Instead of one big search, the system plans, gathers evidence, writes a draft, then fact-checks itself.

1 · Decompose
Break the question into sub-questions
The AI reads the column prompt + the document's section index and makes a retrieval plan.
You are preparing an information-extraction plan for one cell of a matrix. Document: {doc.meta} Available sections: [section index from wiki] User prompt: "Revenue YoY change?" Return JSON: - sub_questions: 2-4 specific retrieval queries - expected_answer_shape: text | number | currency | percentage | list | table - target_sections: up to 3 section ids most likely to contain the answer
2 · Gather
Retrieve evidence for each sub-question
Each sub-question is sent to the retriever (see below). Results are deduplicated by chunk ID.
sub_question 1
→
retriever.retrieve(q, doc, k=6)
→
6 evidence chunks
3 · Draft
Write the answer from evidence only
Hard rule: every factual claim must cite at least one chunk ID. No hallucination allowed.
Answer the user prompt using ONLY the evidence below. Every factual claim must cite at least one chunk id. Return JSON: { answer, citations: [chunk_ids], reasoning_trace: [...] } Prompt: "Revenue YoY change?" Sub-questions: [...] Evidence: [c-42] (p.42, src=wiki.metric) Revenue was $383,285M in FY2023 vs $394,328M in FY2022... [c-43] (p.43, src=chunk.vector) The decrease was primarily driven by lower iPhone sales...
4 · Verify
Fact-check each claim in the draft
For each sentence in the reasoning trace, re-retrieves 3 chunks and asks the AI: "does this evidence actually support this claim?"
Decide whether the following evidence supports, contradicts, or does not contain the claim. Claim: "Revenue grew -2.8% YoY to $383B" Evidence: [3 re-retrieved chunks] Return JSON: { claim, status: "supported"|"contradicted"|"missing", note }
5 · Revise (if needed)
Fix contradicted or unsupported claims
If any claim is contradicted or missing, fetch fresh evidence and re-draft with the verifier's notes. Done once max — then confidence = "low".

Cell result stored + streamed to browser
The final result includes everything — answer, citations with bboxes, full reasoning trace, confidence, tokens used, latency.

⑥ The 3 Retriever Modes (How Evidence is Found)

All three modes implement the same interface. You can switch between them in the UI — the benchmark compares their quality.

🔵 Naive
Simple vector similarity
  • 1 Embed the query into a vector
  • 2 VEC Search LanceDB for nearest k chunks
  • 3 Return top-k by cosine distance
Fast, cheap. But a query like "revenue" might miss chunks that say "net sales" or "total income".
🟡 ISD Retriever
Decompose → vector → LLM rerank
  • 1 LLM Decompose query into 2-5 sub-queries + target sections
  • 2 VEC Vector search per sub-query, boost target sections
  • 3 Deduplicate, take top 30 candidates
  • 4 LLM Attention rerank: score each chunk 0.0–1.0
  • 5 Return top-k by attention score
🟢 Wiki Retriever
Structured knowledge first
  • 1 Search pre-built wiki metrics & claims
  • 2 Find relevant section by questions_answered index
  • 3 Drill into that section's chunks
  • 4 VEC Vector fallback for anything not found above
Highest quality. The wiki already extracted "Revenue = $383B (p.42 c-42)" — no need to search at all.

⑦ ISD Retriever — Its Own Sub-Prompts

The ISD retriever (used inside the Gather step) is itself AI-powered, with 2 LLM calls:

Decompose prompt (inside the retriever)

You are planning retrieval over a long document. Given a user intent and the document's section index, return: - sub_queries: 2-5 specific retrieval queries that together cover the intent - target_section_ids: up to 5 section ids most likely to contain the answer Section index: [s-1] Risk Factors [s-7] Results of Operations ... Intent: "Revenue YoY change"

Attention rerank prompt

Score each chunk's relevance to the query on a STRICT 0.0-1.0 scale. 1.0 = chunk directly answers the query 0.7 = strongly relevant, partial answer 0.4 = tangentially related context 0.0 = irrelevant Return JSON { scores: [{ chunk_id, score }] }. Score every chunk. Query: "Revenue YoY change" Chunks: [c-42] Total net sales were $383,285 million... [c-43] iPhone net sales decreased by $1,636 million... [c-18] The company faces risks from global economy...
