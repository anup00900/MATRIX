Prompt sent per batch (each call sees 5 page images):

You are extracting structured content from pages of a financial filing. For each page image provided, return a JSON array where each element is: { "page_no": int, "text": "full extracted text of the page", "headings": [{"text": str, "level": 1|2|3}], "tables": [{"caption": str, "markdown": str}], "is_mostly_image": bool } [image: page 1] [image: page 2] [image: page 3] [image: page 4] [image: page 5] Return only valid JSON. Do not invent content not visible in the images.
Pros
✓ GPT-4.1 sees actual visual layout — handles multi-column, complex tables natively
✓ No PyMuPDF text extraction at all
✓ Scanned pages work without separate OCR step
✓ Heading detection is visual (font size, bold) — more reliable than heuristics
✓ Tables extracted as proper markdown — not garbled text
Cons
✗ No pixel bboxes (GPT-4.1 can't return coordinates) — citations lose PDF highlight positions
✗ Higher cost vs local PyMuPDF (~8 API calls for 40-page doc)
✗ Slower ingest (sequential batches unless parallelised)

Two-pass: images → raw text, then text → structure

Pass 1 prompt (vision, simple):

Transcribe the text from each page image exactly as it appears. Preserve paragraph breaks. For tables, render as markdown. Return JSON array: [{ "page_no": int, "text": str }] [image: page 1] ... [image: page 5]

Pass 2 prompt (text only, cheap):

Given the full text of a financial filing (pages concatenated), identify section boundaries, headings, and their page numbers. Return JSON: { sections: [{ title, level, page_start, page_end }] } [full text from pass 1]
Pros
✓ Cleaner separation of concerns — easy to debug each step
✓ Pass 2 is text-only (cheap) — most cost is in Pass 1
Cons
✗ Two full LLM passes per doc — more total calls than Approach A
✗ No bboxes (same as A)
✗ Pass 1 misses structural signals (headings visible visually) — Pass 2 can't recover them from plain text
C
One image per API call — fine-grained, parallelised

Parallelism: All 40 page calls fired concurrently (bounded by TPM budget). The Azure TPM budget in jobs/budget.py already handles this — ingest draws from the same token-bucket as wiki builds and cell runs.
Pros
✓ Highest fidelity — each page gets full attention
✓ Parallelised — wall-clock time similar to Approach A
✓ Easy to retry a single failed page
Cons
✗ Most expensive — 40 calls vs 8 for a 40-page doc
✗ Context doesn't span pages — a table that spans p.12–p.13 gets split
✗ Cross-page sections harder to detect

Bbox note (all approaches): GPT-4.1 Vision doesn't return pixel coordinates. To keep PDF citation highlights working, the simplest fix is to use pdf2image only for rendering (no text extraction from PyMuPDF), then map extracted text back to page number — citations become page-level only, not bbox-level. This is a trade-off worth discussing.
